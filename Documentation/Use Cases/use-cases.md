# Use Cases

## 1.Add Passenger
* **Актор:** Пасажир
* **Передумови:** Пасажир ще не зареєстрований.
* **Основний сценарій:**
    1. Користувач надає ПІБ, контакти, дані страховки та результат аналізу.
    2. Система відкриває транзакцію.
    3. Транзакція фіксується.
* **Альтернативні сценарії:**
    * **Помилка валідації:** Якщо користувач з таким ПІБ вже існує, система робить "Rollback", і профіль пасажира не створюється.

## 2.Remove Passenger
* **Актор:** Пасажир.
* * **Передумови:** Профіль пасажира вже існує.
* **Основний сценарій:**
    1. Пасажир ініціює видалення акаунту.
    2. Система каскадно видаляє пов'язані записи страховки, аналізів та багажу.
    3. Основний запис пасажира видаляється з БД.

## 3.Add Data about Passenger
* **Актор:** Пасажир
* **Передумови:** Профіль пасажира вже існує.
* **Основний сценарій:**
    1. Користувач надає дані страхового поліса.
    2. Користувач надає результати медичного аналізу.
    3. Система перевіряє валідність термінів дії.
    4. Записи "Insurance" та "FullHealthAnalysisResult" прив'язуються до "PassengerId".
* **Альтернативні сценарії:**
    * **Невалідні документи:** Якщо термін дії документа вже минув або документу не існує, дані не зберігаються.

## 4.Update Data
* **Актор:** Пасажир.
* **Передумови:** Пасажир вже існує в базі даних.
* **Основний сценарій:**
    1. Пасажир надсилає оновлені контакти або нову версію документів.
    2. Дані оновлюються, версія інкрементується.
* **Альтернативні сценарії:**
    * **Невалідні документи:** Якщо термін дії документа вже минув або документу не існує, дані не зберігаються.

## 5.Book Flight
* **Актор:** Пасажир.
* **Передумови:** Пасажир зареєстрований; рейс існує в розкладі.
* **Основний сценарій:**
    1. Пасажир обирає конкретний рейс.
    2. Система перевіряє медичний допуск.
    3. Система перевіряє наявність вільних місць на кораблі.
    4. Створюється запис у таблиці зв'язку.
* **Альтернативні сценарії:**
    * **Медична відмова:** Якщо стан здоров'я не дозволяє летіти, пасажир не додається на рейс.
    * **Забагато пасажирів:** Якщо корабель вже заповнений максимально, пасажир не додається на рейс.

## 6.Add Baggage to Flight
* **Актор:** Пасажир.
* **Передумови:** Пасажир успішно заброньований на рейс.
* **Основний сценарій:**
    1. Пасажиру надається максимально допустима вага багажу.
    2. Система автоматично знаходить вільний контейнер на цьому рейсі.
    3. Багаж реєструється в обраному контейнері.

## 7.Add Spaceship
* **Актор:** Адміністратор.
* **Передумови:** Корабель з такою назвою ще не зареєстрований.
* **Основний сценарій:**
    1. Адмін вводить назву, дату випуску та ліміти (пасажири/контейнери).
    2. Система перевіряє унікальність назви корабля.
    3. Запис додається до таблиці.
* **Альтернативні сценарії:**
    * **Дублікат:** Система видає помилку, якщо корабель з такою назвою вже є у флоті.

## 8.Add Flight
* **Актор:** Адміністратор.
* **Основний сценарій:**
    1. Адмін створює рейс, вказує дату і пункти відправлення/призначення.
    2. Система перевіряє часову логіку(дата прибуття після дат відправлення).
    3. Створюється рейс із унікальним композитним ключем.

## 9.Manage Flight
* **Актор:** Адміністратор.
* **Основний сценарій:**
    1. Адмін змінює параметри існуючого рейсу, напрклад додає космічний корабель чи змінює дату рейсу.
    2. Система оновлює дані в таблиці рейсу.
* **Альтернативні сценарії:**
    * **Об'єкт не знайдений:** Якщо рейс або новий корабель не існують, повертається помилка.

## 10. Add Container
* **Актор:** Адміністратор.
* **Передумови:** Існує рейс, до якого потрібно додати контейнери; обраний космічний корабель має вільні слоти для контейнерів.
* **Основний сценарій:**
    1. Адмін вказує тип контейнера та його максимальну вантажопідйомність.
    2. Система перевіряє, чи не перевищує загальна кількість уже доданих контейнерів ліміт корабля на цьому конкретному рейсі.
    3. Контейнер реєструється в системі та закріплюється за рейсом.
* **Альтернативні сценарії:**
    * **Перевищення ліміту:** Якщо кількість контейнерів на рейсі вже дорівнює максимальній кількості на кораблі, система видає помилку.
   
### 11. Add Robot
* **Актор:** Адміністратор.
* **Мета:** Реєстрація нового робота в загальну базу активів.
* **Основний сценарій:**
    1. Адмін вказує модель робота, серійний номер та характеристики.
    2. Робот реєструється в системі.

### 12. Add Robot to the Container
* **Актор:** Адміністратор.
* **Передумови:** Робот та контейнер уже існують; контейнер призначений на конкретний рейс.
* **Основний сценарій:**
    1. Адмін обирає робота та ID цільового контейнера на конкретному рейсі.
    2. Система перевіряє вантажопідйомність контейнера (чи не перевищує вага робота залишок вільної ваги).
    3. Робот закріплюється за контейнером для поточної місії.
* **Альтернативні сценарії:**
    * **Перевантаження:** Якщо вага робота завелика для контейнера, додавання відхиляється.
